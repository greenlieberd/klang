# Skill: Pure Data for Embedded Audio

## What it is
Pure Data (Pd) is a visual dataflow programming language for audio and media. In KLANG projects, it's used to design audio patches that are then compiled to C++ for deployment on Daisy Seed via the **Heavy/hvcc** compiler.

## When to use Pd vs C++

| Use Pd | Use C++ |
|---|---|
| Patch-based signal flow thinking | Performance-critical DSP |
| Rapid prototyping of audio algorithms | Full hardware control (ADC, UART, etc.) |
| Generative/modular-style patching | When hvcc compatibility is uncertain |
| Sharing patches with musicians | Custom DSP not in Pd's object set |

## Key Pd objects for audio

**Sources**
- `[osc~ freq]` — sine oscillator
- `[phasor~ freq]` — sawtooth 0–1
- `[noise~]` — white noise
- `[adc~]` — audio input from hardware

**Processing**
- `[vcf~ Q]` — resonant filter (freq from left inlet, bandwidth from right)
- `[bob~]` — Moog ladder filter approximation
- `[clip~ lo hi]` — waveshaper / clipper
- `[delwrite~ name size]` / `[delread~ name delay]` — delay lines
- `[*~]` — multiply (VCA/modulation)
- `[+~]` `[-~]` — add/subtract

**Control**
- `[r name]` / `[s name]` — receive/send (named buses)
- `[metro ms]` — clock/timer
- `[sel value]` — conditional branching
- `[line~]` — smooth interpolation for parameter changes

**Output**
- `[dac~]` — audio output to hardware

## Compiling to Daisy with hvcc

```bash
# Install heavy
pip install hvcc

# Compile a Pd patch
hvcc mypatch.pd -n MyPatch -o ./output -g c

# Output: C++ source that links with DaisySP/libdaisy
# Copy generated source into your Daisy project
```

**Compatibility notes:**
- hvcc supports a subset of Pd — most audio objects work, some GUI/MIDI objects do not
- Test in desktop Pd first, then compile
- hvcc output replaces the AudioCallback — you provide parameter updates from main()
- Check hvcc compatibility list before relying on an exotic object

## Parameter mapping (Pd → hardware)

hvcc generates parameter setters you call from C++:
```cpp
// Generated by hvcc for a [r cutoff] receiver in your patch
Patch_setParameterValue(&patch, HV_PARAM_ID_CUTOFF, value, 0);
```

Map ADC reads to hvcc parameter setters in main loop.

## Gotchas
- `[dac~]` in Pd maps to audio output — in hvcc, this becomes the audio callback output
- `[adc~]` maps to audio input — hardware mic/line in
- No filesystem access in hvcc — no `[readsf~]` or `[writesf~]`
- CPU usage is harder to predict than hand-written C++ — profile after compilation
- hvcc does not support all externals — stick to vanilla Pd objects
